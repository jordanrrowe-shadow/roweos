## Fix 1: Instagram PWA Token Sync

### Problem

Instagram OAuth in mobile Safari writes `roweos_ig_oauth_pending = '1'` to browser localStorage. iOS partitions localStorage between browser and PWA. When user opens the PWA, the visibility change listener (line ~107815) checks this flag -- it's `null` in PWA context, so Firestore sync never triggers. Token exists in Firestore but PWA never reads it.

### Root Cause

The visibility change listener at line ~107810 gates Firestore social token sync behind `roweos_ig_oauth_pending === '1'`. This flag only exists in the browser context where OAuth happened, not in the PWA.

### Fix

**In `index.html` startup social token sync (line ~136080):**
- Remove or relax the guard on the Firestore `social_tokens` query
- Always query `users/{uid}/social_tokens/` on Firebase auth ready
- For each doc found, check if localStorage already has the token -- if not, write it
- This catches: cross-device sync, browser-to-PWA sync, any missed OAuth callbacks

**In the visibility change listener (line ~107810):**
- Remove the `roweos_ig_oauth_pending` guard
- Always check Firestore when app becomes visible and Firebase auth is ready
- Keep a debounce (e.g., don't re-check within 30 seconds) to avoid hammering Firestore

### Files Changed

- `RoweOS/dist/index.html`: startup sync function (~136080), visibility listener (~107810)

## Fix 2: Brand Persistence After "Add Brand" (Bug A)

### Problem

New brands are created and saved to `brands[]` and Firestore, but the app immediately switches back to brand 0. User sees brands "not staying."

### Root Cause

`USER_DATA_KEYS` object (line 62912) has no `selectedBrand` property. `completeBrandSetup()` (line 143661) writes to `localStorage.setItem(USER_DATA_KEYS.selectedBrand, ...)` which resolves to key `"undefined"`. `finalizeOnboarding()` (line 143738) reads from the same broken key, gets nothing, falls back to `'0'`, switches to brand 0.

### Fix

Change two lines to use the correct key `'roweos_selected_brand'` (matching 17 other occurrences in codebase):
- Line 143661: `localStorage.setItem('roweos_selected_brand', newBrandIndex.toString());`
- Line 143738: `var newBrandIdx = parseInt(localStorage.getItem('roweos_selected_brand') || '0');`

### Files Changed

- `RoweOS/dist/index.html`: lines 143661, 143738

## Fix 3: "Add Brand" After LifeAI Skips Web Search Fork (Bug 1)

### Problem

After completing LifeAI onboarding, clicking "+Add Brand" goes straight to Brand Basics (step 7), skipping the "Analyze Website / Create Manually" fork (step 2).

### Root Cause

`launchOnboardingForNewBrand()` (line 144237) does NOT call `resetOnboardingState()`. Leftover state from LifeAI flow (`selectedOnboardingMode = 'life'`, `sharedSelectedProvider` set) causes `proceedFromBetaWelcome()` (line 139725) to route incorrectly:
- If `selectedOnboardingMode === 'life'` -> goes to life steps
- Else -> goes to step 7 (skipping step 2)

### Fix

Add `resetOnboardingState()` as the first call in `launchOnboardingForNewBrand()`, before showing the modal.

### Files Changed

- `RoweOS/dist/index.html`: `launchOnboardingForNewBrand()` (line 144237)

## Fix 4: LifeAI Step 0 Web Import Fork Not Showing (Bug 2)

### Problem

The LifeAI onboarding modal opens but never shows the "Import from Website / Manual Setup" fork (Step 0). Goes straight to step 1 (Basics).

### Root Cause

`openLifeOnboarding()` (line 141969) sets `step0.style.display = ''` which removes the inline style. But if a CSS rule sets `.life-onboarding-step { display: none }`, the element stays hidden. Need explicit `display = 'block'`.

### Fix

- Line 141973: Change `step0.style.display = ''` to `step0.style.display = 'block'`
- Also verify no CSS rule is hiding `.life-onboarding-step` -- if found, add exception for `#lifeOnbStep0` when active

### Files Changed

- `RoweOS/dist/index.html`: `openLifeOnboarding()` (line ~141973), possibly CSS section

## Fix 5: Brand/LifeAI State Bleed (Bug 3)

### Problem

After LifeAI onboarding: brand accent color defaults to blue/purple instead of selected color, brand logo (fleur-de-lis) appears on LifeAI profile sidebar, brand name "Riley" shows as LifeAI profile name.

### Root Cause

Window-level temporary variables persist between onboarding flows:
- `window._onboardingBrandColor` -- brand color bleeds
- `window._onboardingBrandColorLight`
- `window._onboardingLogo` -- brand logo bleeds
- `window._onboardingColorMode`
- `window.onboardingBrandName` -- brand name bleeds
- `window._pendingWebSearchUrl`

`resetOnboardingState()` (line 138316) clears some variables but not the color/logo ones. And `launchOnboardingForNewBrand()` doesn't call it at all (Fix 3 addresses this).

### Fix

Extend `resetOnboardingState()` to also clear:
```
window._onboardingBrandColor = null;
window._onboardingBrandColorLight = null;
window._onboardingLogo = null;
window._onboardingColorMode = null;
window._pendingWebSearchUrl = null;
localStorage.removeItem('roweos_pending_web_search_url');
localStorage.removeItem('roweos_web_import_state');
```

Also add cleanup at the END of `finishLifeAIOnboarding()` to clear these same variables, preventing bleed into subsequent brand onboarding.

### Files Changed

- `RoweOS/dist/index.html`: `resetOnboardingState()` (line 138316), `finishLifeAIOnboarding()` (line ~142904)
